## Post body

I built Corral to answer a question I could never answer honestly about my own code:
*do my tests actually test anything, or do they just pass?*

The usual ways of checking don't help much, because the thing that wrote the code tends to
be the thing that grades it — the model writes the code and then says it's good, or writes
the tests for its own code and reports that they pass. That's the author grading their own
work, and authors are kind to themselves.

Corral works the other way. It's a small headless "brain" that runs an adversarial herd:
one model plants deliberate faults in your code (this is just mutation testing), a
*different* model reads your test suite, and the brain itself **runs your tests against
every planted fault in a sandbox** and signs the result. The judge is never the author,
and nothing is taken on a model's word — it's all executed.

Here's the example that made me trust it. A password validator: strong only if it's ≥12
characters *and* has an uppercase, a lowercase, a digit, and a symbol. And a test that any
of us might write — it checks that a short password fails and a good one passes:

```
Corral planted goal-violating variants of that validator and ran the test against each:

  Python      2/7 faults caught   →  5 slipped through
  Ruby        2/8                 →  6
  JavaScript  1/8                 →  7
  TypeScript  0/6                 →  6
```

The test never checks the four character-class rules at all, so a variant that quietly
drops "must contain a digit" passes it every time. Same blind spot, in every language,
~40 seconds each. It's the kind of gap that's genuinely easy to miss — I've shipped it.

The part I'm actually proud of: when your suite has a hole, the herd **writes the test you
were missing and hands it back to you.** It's not there to scold; it's there to give you
the assertion you didn't think to write. And it's fair — pointed at a genuinely thorough
suite (Corral's own SQL guard, in Go) it killed every planted fault and signed it *clean*.
A gate that only ever finds fault is theatre.

### How it works, briefly

- **Mutation testing, measured by execution.** "Do your tests catch a deliberately broken
  version of the code?" is a decades-old, honest measure of suite adequacy. Corral's twist
  is that the mutants are *semantic* (drop a whole rule), generated by an LLM, and the kill
  rate is measured by actually running the tests — never a model's self-report.
- **Decorrelation is enforced.** The model that critiques the suite must differ from the
  one that wrote the exposing test; the run refuses to start otherwise.
- **Certify by execution.** If the toolchain to run your tests isn't present, it fails
  closed — it won't guess. Five languages today (Go, Python, Ruby, JS, TS), each a small
  plugin.
- Every verdict is a hash-linked, tamper-evident record. A stranger can re-run it.

### Honest about the seams

Two field notes instead of overselling. One is called **"Your tests suck"** — and I mean
that the way Roger Ebert meant *Your Movie Sucks*: it's written by someone who loves good
tests and wants yours to be good too. The other, **"Good baking means always mind the
D'oh,"** is a list of the mistakes I made building this — for a while the gate could only
run Go, so I was quietly re-testing Go while telling a five-language story; a `--user` pip
install landed where the sandbox couldn't see it; a `_js` filename got silently dropped by
a Go build constraint. Building a *certifier* is humbling in a way building a *builder*
never was: a builder is bounded by what a model can write, a certifier by what its sandbox
can actually run.

It's Elastic-2.0, runs on your own keys, one Go binary plus a couple of worker processes.
corralai.dev has the repo and the field notes. I'd genuinely love feedback — especially on
mutation-testing prior art, the decorrelation guard, and where an execute-don't-report gate
*can't* help (exploratory pentesting is attributed, not certified — I try to be clear about
that seam).

## First-comment prep (warm, not defensive)

- "Isn't this just mutation testing?" — Yes, and that's the honest, load-bearing part. The
  additions are (1) enforced decorrelation across models and (2) a signed, re-runnable
  record — mutation testing you can hand to someone else and they can check.
- "Why an LLM for the mutants?" — For *semantic* faults (drop a rule) and for writing the
  missing test. The scoring stays deterministic execution.
- "The one-command competitor does X" — fair; Corral's frictionless one-command story is
  the next thing I'm building. Today it's a brain plus workers.
