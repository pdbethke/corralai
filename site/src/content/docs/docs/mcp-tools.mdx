The corral brain is **MCP-native**: `corral` serves `/mcp/` (and `/healthz`) on
`$CORRALAI_ADDR`. Every audit action ‚Äî certifying a build, staging a control,
claiming and running work in the jail, reading and writing shared knowledge ‚Äî
is an MCP tool call to the brain, which records it as an attributed side
effect. An MCP-native client (an agent, an operator's CLI, an IDE) points its
`.mcp.json` at the brain's endpoint and gets the tools below.

Access is enforced per call: workers get the coordination + work tools;
admin/superuser tools are gated; some (like sharing a recording) are
human-gated and refuse delegated subagent tokens. This is a guide, not the wire
spec ‚Äî the authoritative schema is whatever the running brain advertises via
`tools/list`.

## Certify ‚Äî the audit ingest + the adversarial pool

- **`report_build`** ‚Äî `corral certify`'s ingest endpoint: turn a raw build
  record (repo, commit, command, exit code) into a signed, tamper-evident,
  stored accountability record.
- **`start_adversarial_run`** ‚Äî *(admin)* start an adversarial-testing-pool
  run against a code + dev-test pair ‚Äî mutant-generator, test-writer, and
  test-critic roles, decorrelation-enforced, gated by the brain-side adequacy
  jail, certified via the signed record chain. What `corral certify --local`
  runs in-process; this is the hosted, brain-mediated equivalent.
- **`get_adversarial_run`** ‚Äî *(admin)* query a run's status and, once
  converged, its signed verdict, by the id `start_adversarial_run` returned.

## The control gate ‚Äî vetted, org-owned checks

A control is an authored test scored for mutation-adequacy against a goal,
staged unvetted, then human-promoted into the set the control gate actually
runs ‚Äî see [Trust & security](/docs/concepts/trust-and-security/) for the
human-gate rule these tools sit behind.

- **`import_control_bundle`** ‚Äî *(admin)* import a control-standard bundle
  (e.g. `asvs-l1`) as owner-scoped goals.
- **`list_control_goals`** ‚Äî list your control goals: the bar the control
  gate holds code to.
- **`stage_control`** ‚Äî *(admin)* author a candidate test for a goal+target
  and score its mutation-adequacy. Stored **unvetted** ‚Äî never gates until
  `promote_control`.
- **`list_pending_controls`** ‚Äî list your unvetted candidates awaiting
  review (goal, target, kill rate).
- **`get_control`** ‚Äî fetch one pending candidate in full: test source, kill
  rate, surviving mutants, reviewer triage ‚Äî to read before promoting.
- **`promote_control`** ‚Äî *(admin)* approve a pending candidate into the
  vetted store the control gate runs. The recorded, attributed human gate.
- **`reject_control`** ‚Äî *(admin)* delete a candidate control, vetted or not.

## Coordination ‚Äî join the herd, claim work

- **`bootstrap`** ‚Äî enter a coordination session in one call: register yourself and get active peers, your live claims, and recently-completed work. Call this first.
- **`heartbeat`** ‚Äî refresh your presence (and optionally report status: `working` / `awaiting_approval` / `idle`).
- **`claim_paths`** ‚Äî lease files/dirs/branches. Exclusive claims are *enforced* ‚Äî always check `granted` before acting; pass `exclusive:false` for an advisory lease.
- **`release_claims`** ‚Äî release your leases (specific paths, or all of yours).
- **`whois`** / **`list_active`** ‚Äî one agent's profile + active claims, or every agent that's heartbeat within the window.
- **`mark_done`** / **`coordination_status`** ‚Äî record finished work so peers don't duplicate it, or snapshot active agents + live claims + recent work in one call.
- **`fleet_claims`** ‚Äî see who across the fleet holds a lease on a subject (cross-brain, cryptographically verified, advisory-only).
- **`spawn_subagent`** / **`despawn_subagent`** / **`list_subagents`** ‚Äî spawn a governed out-of-process worker under a scoped, TTL-bound delegation token, tear one down, or list active ones.

## The task queue ‚Äî gated work inside a jail run

These back the audit runs above (adversarial-pool task DAGs, repo/control
gate work) ‚Äî a task queue with a verify gate, not a directive-to-build
pipeline. See [The task queue + verify gate](/docs/concepts/queue-and-verify/).

- **`claim_task`** / **`complete_task`** ‚Äî pull the next eligible task in dependency order; `complete_task` refuses to close a **gated** task (one with a `Verify` command) until a matching passing execution is on record.
- **`report_finding`** / **`list_findings`** / **`resolve_finding`** ‚Äî file a vuln/bug/design finding with a severity, list open ones, or mark one resolved.
- **`list_tasks`** / **`cancel_task`** / **`reopen_task`** / **`retarget_dependencies`** / **`supersede_task`** / **`enqueue_task`** ‚Äî inspect and reshape a run's task graph.
- **`report_execution`** ‚Äî record a command execution (the attributed action ledger + the replay console read from these).

## Missions ‚Äî read-only tracking, not a build pipeline

Every audit run (adversarial pool, repo-work) is tracked as a mission record
internally, so the same durable history + replay applies to it ‚Äî there is no
tool that turns a directive into a mission to build; `create_mission` doesn't
exist. These are the observability surface over that tracking record:

- **`mission_status`** / **`list_missions`** ‚Äî one run's status, or recent runs and theirs.
- **`resolve_review`** / **`pause_mission`** / **`resume_mission`** / **`cancel_mission`** ‚Äî operator controls over a run's lifecycle.
- **`mission_history`** ‚Äî past runs: a list (status, duration, counts, what-got-learned) or, given an id, the full drill-down.
- **`mission_replay`** ‚Äî replay one run's recorded event stream, oldest-first.
- **`mission_analytics`** ‚Äî analyze the run event log (DuckDB): a named report (`missions` / `agents` / `kinds` / `findings` / `model_comparison`) or an ad-hoc read-only `SELECT`.

## Working in a repo ‚Äî the jail ‚Üí PR path

- **`repo_snapshot`** ‚Äî snapshot the current run's working copy.
- **`repo_push`** ‚Äî hand code back through the brain (which holds the credential) as a reviewed PR ‚Äî the only way code leaves the jail.
- **`read_repo`** / **`repo_tree`** / **`repo_grep`** ‚Äî read a file, list files, or literal-search across the working copy.
- **`repo_search`** ‚Äî semantic/hybrid code search over the working copy; returns `path:line` ranges ranked by meaning.
- **`sync_head`** / **`sync_pull`** / **`sync_put`** / **`sync_delete`** ‚Äî the artifact-sync surface for exchanging files with a run's working copy outside the snapshot/push path (`sync_put`/`sync_delete` are human-gated).

## Headless browser ‚Äî web app automation & verification

Built-in headless browser tools powered by Go-native `rod` allowing agents to interact with and verify running web applications directly:

- **`browser_navigate`** ‚Äî navigate the browser session to a target URL.
- **`browser_click`** ‚Äî simulate clicking an element on the page matched by CSS selector.
- **`browser_input`** ‚Äî enter text into a text input element matched by CSS selector.
- **`browser_get_html`** ‚Äî retrieve the current raw HTML source of the active page.
- **`browser_screenshot`** ‚Äî capture a full-page PNG screenshot of the current page and save it directly as a task artifact. Gated: the calling agent must hold the active task lease claim.
- **`save_task_artifact`** / **`list_task_artifacts`** ‚Äî save an artifact (e.g. a screenshot) to the database directly, or list a run's saved artifacts.

## Knowledge ‚Äî memory & reference corpus

- **`search_memory`** / **`get_memory`** / **`list_memory`** ‚Äî search (BM25), fetch, or list the memory you can see (your own + the shared team knowledge base).
- **`add_memory`** ‚Äî persist a memory entry. Private by default (needs a memory owner); `shared=true` (the team knowledge base) needs an admin.
- **`promote_memory`** ‚Äî *(admin)* share an existing entry team-wide, or make it private again.
- **`add_reference`** / **`search_reference`** / **`list_references`** ‚Äî ingest a URL or raw text into the reference corpus, semantically search it, or list sources. Hits are fenced as **untrusted** data with provenance.
- **`promote_reference`** ‚Äî *(admin)* mark a reference source vetted/trustworthy.

## The learning loop ‚Äî human-gated

- **`list_proposals`** / **`approve_proposal`** / **`reject_proposal`** ‚Äî list the learning loop's clustered findings ‚Üí proposed skills, and act on one. Approving is what turns a run's lessons into authoritative, reusable knowledge; both writes sit behind the human gate.

## Recordings ‚Äî export & share replays

- **`list_recordings`** ‚Äî list exported replay recordings (ACL-scoped; dev mode open).
- **`query_recordings`** ‚Äî analyze recordings in DuckDB (named report or superuser ad-hoc read-only SQL).
- **`get_recording_replay`** ‚Äî return one recording's replay event stream by slug or mission id.
- **`share_recording`** ‚Äî share a recording with a team catalog or everyone. **Human-gated** ‚Äî delegated subagent tokens are refused.

## Topology & the console ‚Äî observability

- **`report_host`** ‚Äî announce your runtime facts (host, model, jail) so the topology view can map where every agent runs.
- **`swarm_topology`** ‚Äî the live topology: each active worker with its host, model, jail, and any model-drift from the configured role policy.
- **`report_activity`** ‚Äî report a tool call you just made so the live console shows what every agent is doing, durably recorded for replay when a mission id is present.
- **`report_thought`** ‚Äî report a piece of your own reasoning, verbatim, for the story/recording engine. Opt-in per run (off by default); never rewritten or summarized by the brain.
- **`send_instruction`** / **`check_instructions`** / **`ack_instruction`** ‚Äî the operator‚Üîagent inbox: send a nudge, read yours, acknowledge one.

## The fleet oracle

- **`ask_fleet`** ‚Äî a natural-language question about the whole fleet's state (missions, tasks, telemetry across all swarms). Read-only; the generated SQL runs inside a locked-down DuckDB connection ‚Äî see [Trust & security](/docs/concepts/trust-and-security/).

## The MCP gateway ‚Äî share *reach*

Register any upstream MCP server with the brain; the herd then shares
*capability* while the brain holds the *credential*.

- **`register_endpoint`** ‚Äî register your personal upstream MCP server (the `auth_token` is held by the brain, never returned).
- **`list_my_endpoints`** / **`list_all_endpoints`** ‚Äî list yours (or, *admin*, every registered upstream, owner + public flag, no secrets).
- **`promote_endpoint`** ‚Äî *(admin)* promote a personal endpoint to public (team-wide or scoped), optionally swapping in a team credential.
- **`set_endpoint_enabled`** / **`remove_endpoint`** ‚Äî enable/disable or delete an endpoint you own (or any, as admin).
- **`list_capabilities`** ‚Äî the upstreams you may use and their tools. Discover here, then call.
- **`call_capability`** ‚Äî call a tool on an upstream you may use, proxied and audited by the brain.

## Identity & administration ‚Äî superuser

- **`whoami`** ‚Äî who the brain sees you as: your verified principal, superuser flag, and whether you're allowed to use this brain.
- **`create_superuser`** ‚Äî create/promote a superuser (the *first* is free to bootstrap; after that, superuser-only).
- **`list_principals`** / **`add_member`** / **`set_superuser`** / **`remove_principal`** ‚Äî *(superuser)* list everyone, allow a member, flip a superuser flag, or revoke access.
- **`mint_observer`** ‚Äî *(superuser)* mint a **read-only** observer token: the holder can watch the swarm but cannot act (no claims, instructions, or tool calls).
